# 支付工作使用说明

**文档版本**: v1.1  
**创建日期**: 2026-03-15  
**适用对象**: 经办人、复核人、审批人、财务人员、系统管理员

---

## 一、功能简介

“支付工作”用于把已具备发放条件的数据组织成支付计划，并完成后续复核、审批、财务提交、银行结果导入、失败反馈与重发处理。

支付工作的起点是“支付计划生成”，终点不是“提交银行”，而是：

- 发放成功的记录完成确认
- 发放失败的记录进入反馈、更正和重发
- 批次能够形成可追溯的闭环结果

当前系统采用的是：

- `Excel 导出` 给银行或财务线下处理
- `Excel 导入` 回写银行发放结果

当前系统不包含：

- 直连银行接口
- 自动扣款或自动回盘

---

## 二、适用角色与权限

### 2.1 角色分工

| 角色 | 主要操作 |
|------|----------|
| 经办人 | 生成支付计划、查看明细、创建批次、处理驳回、接受失败反馈、信息更正、发起重发 |
| 复核人 | 复核支付批次、查看统计和明细、通过或驳回 |
| 审批人 | 审批支付批次、查看统计和明细、通过或驳回 |
| 财务人员 | 查询待财务批次、导出银行文件、提交银行、导入结果、确认失败台账 |
| 系统管理员 | 配置菜单权限、字典和导入导出模板，处理异常账号与权限问题 |

### 2.2 建议权限

- 支付计划生成：`shebao:payment:plan:*`
- 支付计划复核：`shebao:payment:review:*`
- 支付计划审批：`shebao:payment:approve:*`
- 财务批次/银行发放：`shebao:finance:batch:*`、`shebao:finance:bank:*`
- 失败处理：`shebao:finance:failure:*`

### 2.3 使用建议

- 经办人至少需要具备支付计划查询、生成、批次创建权限
- 复核人和审批人应只具备查看与审核权限，不应具备经办生成权限
- 财务人员应具备银行文件导出、提交银行、导入结果权限
- 测试环境若按钮可见但接口报无权限，应优先排查按钮权限与接口权限是否一致

## 四、工作闭环说明

### 4.0 测试前准备

在当前测试环境中，支付闭环功能依赖最新表结构。开始联调前，必须先完成数据库升级。

执行要求：

1. 先备份测试库 `lfpm`。
2. 执行脚本 `sql/payment_closure_minimal_upgrade_20260315.sql`。
3. 确认 `benefit_determination`、`shebao_subsidy_distribution`、`distribution_batch` 三张表已补齐新字段。
4. 重启后端服务后再进入支付页面测试。

该脚本主要作用：

- 补齐支付闭环主链路需要的最小字段
- 回填 `payment_plan_generated`
- 统一旧数据中的 `subsidy_type` 口径
- 为支付计划、批次、财务和失败处理联调提供基础运行条件

支付工作建议按以下顺序执行：

1. 经办人生成支付计划。
2. 经办人确认明细并创建支付批次。
3. 复核人审核支付批次。
4. 审批人审批支付批次。
5. 财务人员导出银行 Excel 并线下提交。
6. 财务人员导入银行结果 Excel。
7. 系统将结果拆分为成功和失败记录。
8. 经办人处理失败反馈并发起二次/三次发放。

闭环判断标准：

- 批次内全部记录都能落到“成功”“失败待处理”“人工处理”之一
- 失败记录能追溯原批次、原失败原因、原重发次数

## 五、操作步骤

### 5.1 生成支付计划

适用角色：经办人

1. 进入 `支付计划生成` 页面。
2. 选择补贴类型、发放月份、街道办等条件。
3. 点击“预览统计”，确认总人数和总金额。
4. 点击“生成计划”。
5. 系统生成待处理支付记录。

操作要点：

- 生成前先确认本月是否已生成过同类支付数据
- 若测试环境刚完成数据库升级，先做一次“预览统计”确认基础数据已可读取
- 若是从预到龄通知流程进入，优先按通知批次生成
- 生成后应及时核对是否存在重复、缺失或异常金额

### 5.2 创建支付批次

适用角色：经办人

1. 在生成后的记录列表中勾选本次要提交审核的数据。
2. 点击“创建批次”。
3. 系统生成支付批次号。
4. 确认批次进入待复核状态。

建议核对字段：

- 批次号
- 补贴类型
- 发放月份
- 总人数
- 总金额
- 批次状态是否已进入 `pending_review`

### 5.3 支付计划复核

适用角色：复核人

1. 进入 `支付计划复核` 页面。
2. 按批次号或默认待复核条件查询。
3. 查看批次统计信息。
4. 抽查明细，重点关注姓名、身份证号、银行账号、发放金额。
5. 输入复核意见并点击“通过”或“驳回”。

复核重点：

- 是否有重复人员
- 是否有应停发但仍进入计划的数据
- 金额是否异常
- 银行信息是否完整

### 5.4 支付计划审批

适用角色：审批人

1. 进入 `支付计划审批` 页面。
2. 查询待审批批次。
3. 查看统计和抽查明细。
4. 输入审批意见。
5. 点击“通过”或“驳回”。

通过后结果：

- 批次进入待财务

驳回后处理：

- 返回业务侧重新检查、修正后再次提交

### 5.5 财务导出银行文件

适用角色：财务人员

1. 进入 `财务管理 -> 银行发放` 或 `财务管理 -> 批次管理`。
2. 查询 `待财务` 状态批次。
3. 打开目标批次。
4. 点击“生成文件”或“生成银行文件”。
5. 下载 Excel 或银行格式文件。

使用建议：

- 导出后先本地抽查关键字段
- 文件名建议保留批次号，便于结果回盘时匹配

### 5.6 提交银行

适用角色：财务人员

1. 在财务页面找到目标批次。
2. 点击“提交银行”。
3. 系统记录提交动作。
4. 线下将文件提交给银行或财务处理人员。

注意：

- 当前版本中“提交银行”更多是业务动作登记，不代表系统已与银行完成自动交互
- 提交后应保留线下发送凭证或记录

### 5.7 导入银行结果

适用角色：财务人员

1. 银行返回结果 Excel 后，进入财务页面。
2. 找到已提交银行的批次。
3. 点击“导入结果”。
4. 上传银行返回的 Excel。
5. 系统按批次号、人员、账号等信息匹配结果。
6. 匹配成功的记录更新为成功或失败。

导入完成后应立即检查：

- 成功人数
- 失败人数
- 是否存在未匹配记录
- 失败原因是否齐全

### 5.8 失败反馈与信息更正

适用角色：经办人

1. 进入 `财务管理 -> 失败处理` 页面。
2. 按批次号、姓名、失败原因筛选。
3. 查看失败记录。
4. 点击“信息更正”。
5. 修正银行卡、账户名、身份信息或补贴信息。
6. 保存更正说明。

适用场景：

- 银行卡号错误
- 户名不匹配
- 身份证号不匹配
- 银行系统退票

### 5.9 二次发放与三次发放

适用角色：经办人、复核人、审批人、财务人员

1. 经办人在失败处理页面选择需要重发的记录。
2. 发起“重新发放”。
3. 系统生成二次或三次发放批次。
4. 重发批次重新进入复核、审批、财务、结果导入流程。

处理原则：

- 首次失败生成二次批次
- 二次失败生成三次批次
- 多次失败后转人工处理，不建议无限重发

## 六、当前业务口径

### 6.1 支付工作的起点

支付工作的正式起点是：

- `支付计划生成`

不是：

- 财务提交银行
- 银行发放结果导入

因为财务环节必须建立在已完成业务审核的支付批次之上。

### 6.2 闭环口径

只有满足以下条件，才能认为一个支付批次完成闭环：

- 业务侧已生成并审核通过
- 财务侧已提交银行
- 银行结果已导入
- 失败记录已进入补偿流程

### 6.3 当前系统现实情况

当前版本中，支付相关功能并非全部已完整落地，主要体现在：

- 页面已存在，但部分页面字段仍在按联调口径持续收口
- 前后端主链路接口已补齐，仍存在展示字段和异常分支待继续稳定
- 财务导入结果和失败重发已经进入可测试状态，但还不应视为正式投产能力

因此使用时应把它视为：

- `可联调主流程`
- `可推进闭环测试`
- `待继续收口页面字段、模板和异常规则`

而不是：

- 完全可上线的正式支付系统

## 七、注意事项

### 7.1 关于批次

- 批次号应作为对账、导出、回盘、失败追溯的统一标识
- 同一批次应保持人员范围稳定，避免导出后再修改

### 7.2 关于导出文件

- 导出的银行文件应保留原始版本
- 不建议导出后手工大规模修改，否则导入结果时可能无法准确匹配

### 7.3 关于导入结果

- 导入前应确认结果文件属于正确批次
- 若结果文件中存在系统外记录，应先清洗再导入

### 7.4 关于失败处理

- 更正后再重发，不建议直接重复提交失败原记录
- 三次失败后应转人工，不宜继续自动循环

### 7.5 关于职责边界

- 业务人员负责数据正确性
- 财务人员负责提交与结果确认
- 系统管理员不直接替代业务审批和财务确认

## 八、常见使用场景

### 8.1 月度正常发放

建议方式：

1. 经办生成支付计划
2. 复核通过
3. 审批通过
4. 财务导出文件并提交银行
5. 导入结果完成确认

### 8.2 预到龄衔接支付

建议方式：

1. 预到龄通知批次完成核定和复核
2. 审核通过后按通知批次生成支付计划
3. 后续按支付闭环正常处理

### 8.3 银行退票后重发

建议方式：

1. 财务导入结果
2. 经办处理失败记录
3. 更正账号或身份信息
4. 发起二次发放
5. 重新走审核和财务链路

## 九、异常与处理建议

### 9.1 页面有菜单但点了报错

优先排查：

1. 是否缺少对应接口权限
2. 前后端接口路径是否一致
3. 是否已先执行 `sql/payment_closure_minimal_upgrade_20260315.sql`
4. 是否是当前版本尚未完成的占位功能

### 9.2 生成支付计划后没有形成闭环

优先排查：

1. 是否只生成了记录，没有创建批次
2. 是否没有走复核、审批流程
3. 当前库结构是否仍落后于代码口径
4. 财务提交银行和结果导入是否仍有异常分支未覆盖

### 9.3 导入银行结果失败

优先检查：

1. Excel 格式是否符合系统要求
2. 批次号是否匹配
3. 人员、账号、金额是否能在系统中定位到对应明细
4. 当前批次是否已完成数据库结构升级后的重新生成

### 9.4 失败记录无法重发

优先检查：

1. 是否已先完成信息更正
2. 当前记录是否超过可自动重发次数
3. 系统是否已建立失败记录到重发批次的关联

### 9.5 看得到按钮但没有实际结果

这是当前版本可能存在的真实情况。  
应重点判断：

- 页面按钮是否只是前端占位
- 后端接口是否返回固定成功消息
- 是否真正有数据库状态变化

## 十、测试使用建议

为了稳定验证支付闭环，建议按以下方式测试：

1. 先准备一批可正常发放的数据，再准备一批故意失败的数据。
2. 分开记录“页面可操作”“接口返回成功”“数据库状态变化”“闭环已走通”四类结果。
3. 至少覆盖一次首次发放成功和一次失败后二次发放。
4. 对财务提交和结果导入，单独保留测试 Excel 文件。

---

## 十一、总结

支付工作不是单个页面功能，而是一条跨角色、跨页面、跨状态的业务链。

当前版本最重要的使用原则是：

- 以支付计划作为起点
- 以批次审核作为主线
- 以 Excel 导出/导入作为财务确认方式
- 以失败反馈和重发作为闭环补偿机制

最终使用口径为：

`计划生成只是开始，结果确认和失败重发完成后，支付工作才算闭环。`
