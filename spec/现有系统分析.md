## 一、系统概述

### 1.1 项目简介

廊坊社保管理系统是一个基于若依(RuoYi) v3.9.0框架开发的县级养老补贴发放系统,采用前后端分离架构。系统主要管理各类补贴对象的身份认定、补贴申请、审核发放等全流程业务。

## 三、核心业务功能

### 3.1 补贴类型管理

系统支持**4种补贴类型**,每种补贴类型对应独立的数据表:

| 补贴类型 | 类型码 | 数据表 | 核心字段 |
|---------|--------|--------|----------|
| **失地居民补贴** | 1 | shebao_land_loss_resident | 征地时间、认定时间、完成补偿时间 |
| **被征地居民补贴** | 2 | shebao_expropriatee_subsidy | 征地批次、基准日、补贴年限、补贴金额 |
| **拆迁居民补贴** | 3 | shebao_demolition_resident | 拆迁时间、拆迁原因 |
| **村干部补贴** | 4 | shebao_village_official | 任职年限、违法记录 |

### 3.2 补贴发放流程

#### 3.2.1 发放状态流转

```
提交审核 → 待审核(1) → 审核通过 → 待发放(2) → 已发放(4)
                              ↓
                         审核拒绝 → 已拒绝(3)
```

#### 3.2.2 发放状态说明

| 状态码 | 状态名称 | 说明 |
|--------|----------|------|
| 0 | 未发放 | 初始状态 |
| 1 | 待审核 | 已提交,等待审核员审批 |
| 2 | 待发放 | 审核通过,等待财务人员发放 |
| 3 | 已拒绝 | 审核或发放环节被拒绝 |
| 4 | 已发放 | 已完成发放 |

#### 3.2.3 审核操作类型

| 操作类型 | 类型码 | 说明 |
|----------|--------|------|
| 提交审核 | 1 | 新增发放记录后的初始状态 |
| 审核通过 | 2 | 审核员同意发放申请 |
| 审核驳回 | 3 | 审核员或财务人员拒绝 |
| 发放 | 4 | 财务人员完成发放 |
| 重新提交 | 5 | 被拒绝后重新提交申请 |

### 3.3 组织机构管理

#### 3.3.1 架构设计

```
街道办(一级)
  └── 村委会(二级)
       └── 居民(三级)
```

#### 3.3.2 用户编码规则

用户编码采用三级编码格式: `街道办编码-村委会编码-序列号`

示例: `001-002-0001`
- `001`: 银河南路街道
- `002`: 第二村
- `0001`: 该村的第1个居民

### 3.4 权限体系

#### 3.4.1 补贴发放模块权限

| 权限标识 | 权限名称 | 说明 |
|----------|----------|------|
| shebao:subsidyDistribution:list | 查看列表 | 查询发放记录 |

#### 3.4.2 动态权限控制

系统实现了基于状态的动态权限检查:

```java
// 待审核状态的拒绝需要 approve 权限
if ("1".equals(distributionStatus)) {
    if (!SecurityUtils.hasPermi("shebao:subsidyDistribution:approve")) {
        throw new ServiceException("没有权限执行此操作");
    }
}
// 待发放状态的拒绝需要 distribute 权限
else if ("2".equals(distributionStatus)) {
    if (!SecurityUtils.hasPermi("shebao:subsidyDistribution:distribute")) {
        throw new ServiceException("没有权限执行此操作");
    }
}
```

## 四、技术实现特点

### 4.1 数据库设计特点

1. **逻辑删除**: 所有业务表统一使用 `del_flag` 字段实现逻辑删除(0正常, 2删除)
2. **审计字段**: 统一包含 `create_by`, `create_time`, `update_by`, `update_time`
3. **状态管理**: 统一使用 `status` 字段(0正常, 1停用)
4. **外键约束**: 关联表使用外键约束,保证数据完整性
5. **索引优化**: 关键查询字段建立索引,提升查询性能
6. **唯一约束**: 身份证号、用户编码等关键字段设置唯一约束

### 4.2 后端架构特点

1. **分层清晰**: Controller → Service → Mapper,职责明确
2. **MyBatis-Plus**: 使用增强版MyBatis,简化CRUD操作
3. **统一返回**: 使用 `AjaxResult` 统一封装返回结果
4. **权限注解**: 使用 `@PreAuthorize` 注解实现方法级权限控制
5. **操作日志**: 使用 `@Log` 注解自动记录操作日志
6. **参数验证**: 使用 `@Validated` 和 JSR303 注解实现参数验证
7. **逻辑删除配置**: MyBatis-Plus全局配置逻辑删除

```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: delFlag
      logic-delete-value: 2
      logic-not-delete-value: 0
      id-type: auto
```

### 4.3 前端架构特点

1. **Vue 2.x + Element UI**: 成熟稳定,组件丰富
2. **Vuex状态管理**: 全局状态共享
3. **Axios拦截**: 统一处理请求/响应
4. **权限指令**: `v-hasPermi` 实现按钮级权限控制
5. **动态路由**: 基于权限动态加载菜单
6. **表单验证**: 使用Element UI的表单验证规则

#### 4.4.2 补贴发放审核

```java
@Override
public int approveDistribution(Long id, String remark) {
    SubsidyDistribution distribution = this.getById(id);
    if (distribution == null) {
        throw new ServiceException("发放记录不存在");
    }

    // 检查状态是否为待审核
    if (!"1".equals(distribution.getDistributionStatus())) {
        throw new ServiceException("只有待审核状态的记录才能审核");
    }

    // 更新状态为待发放
    distribution.setDistributionStatus("2");
    distribution.setReviewRemark(remark);
    boolean result = this.updateById(distribution);

    if (result) {
        // 记录审核历史
        DistributionReview review = new DistributionReview();
        review.setDistributionId(id);
        review.setOperationType("2"); // 审核通过
        review.setOperationRemark(remark);
        review.setOperatorName(SecurityUtils.getUsername());
        review.setOperationTime(new Date());
        distributionReviewService.insertDistributionReview(review);
    }

    return result ? 1 : 0;
}
```
