# 本地自动化测试最佳实践

## 1. 文档定位

本文档不是某一个业务功能的测试说明，而是当前项目可复用的本地自动化测试最佳实践手册。它面向开发、测试、实施、运维协作场景，目标是沉淀一套团队统一可执行的方法，解决以下问题：

- 本地如何快速搭建可复现的测试环境
- 测试环境如何设计主备方案，避免因为环境故障阻塞验证
- 后端接口、数据库、前端 UI 如何组成完整自动化测试链路
- Playwright 如何尽量模拟真实用户行为，覆盖完整流程而不是只做页面点点点
- 如何产出统一的测试报告，供团队复盘和持续集成使用

“预到龄通知”只是本文中的一个示例，不是本文的业务边界。

---

## 2. 总体方法论

当前项目最适合采用的是“三层自动化 + 两套环境 + 一份统一报告”的方案。

### 2.1 三层自动化

- `构建层`
  - 目标：发现编译错误、依赖问题、基础配置错误
  - 工具：`Maven`
- `接口与数据层`
  - 目标：验证真实后端接口、状态流转、数据库落库结果
  - 工具：`Node.js + fetch + mysql2 + SQL fixture`
- `UI 端到端层`
  - 目标：验证页面交互、角色权限、跨页面流程、关键用户路径
  - 工具：`Playwright + Cursor 内置浏览器`

### 2.2 两套环境

- `主测试环境`
  - 面向真实业务链路验证
  - 依赖真实后端、真实数据库、真实 Redis、真实文件上传目录
- `备测试环境`
  - 面向环境故障兜底、页面开发自测、局部模块隔离验证
  - 可采用前端 mock、局部接口桩、最小数据集

### 2.3 一份统一报告

无论是接口自动化还是 UI 自动化，最终都应该落成统一结构的测试报告，至少包括：

- 测试范围
- 环境信息
- 用例执行情况
- 失败详情
- 截图/Trace/视频/日志位置
- 风险结论

### 2.4 Cursor 内置浏览器的角色

`Cursor 内置浏览器` 应作为正式测试工具纳入计划，而不是临时排障工具。

建议定位如下：

- `Playwright`
  - 负责批量化、可重复执行的 UI 自动化回归
- `Cursor 内置浏览器`
  - 负责真实页面补证
  - 负责 Playwright 失败后的快速复核
  - 负责复杂交互、登录态、路由跳转等场景的人工辅助验证

推荐原则：

- 自动化优先用 `Playwright`
- 当 Playwright 因导航、登录态、浏览器依赖或控件稳定性问题失败时，立即使用 `Cursor 内置浏览器` 补证
- 报告中应明确标注哪些结论来自 Playwright，哪些结论来自 Cursor 内置浏览器

## 3. 测试环境主备方案

## 3.1 主测试环境

主测试环境用于验证完整真实链路，必须满足以下条件：

- 后端可正常启动
- 数据库结构已升级到当前代码版本
- Redis 可用
- 文件上传目录可写
- 前端连接真实后端
- 测试账号、角色、权限数据完整

推荐用途：

- 需求联调
- 缺陷复现
- 闭环回归
- 上线前验收

主环境建议基线：

- 后端：`http://localhost:8087/api`
- 前端：`http://localhost:81`
- 数据库：当前 `.env` 所指定实例
- 文件目录：`PROFILE_PATH`
- 日志目录：`LOG_PATH`

主环境成功标准：

- 能正常登录真实账号
- 能走完真实上传、审批、导出、支付计划等关键链路
- 接口返回与数据库状态一致

## 3.2 备测试环境

备测试环境不是“备用服务器”，而是团队在主环境不可用时仍能继续工作的降级方案。

推荐设计为两种备份模式：

### 模式 A：前端 Mock 备环境

特点：

- 不依赖真实后端
- 用于菜单、路由、权限、表单展示、基础交互验证
- 启动快，适合页面开发阶段

当前项目中，`ruoyi-ui/e2e/login-flow.spec.js` 就属于这一类能力。

适用场景：

- 后端未启动
- 接口尚未开发完成
- 需要快速验证权限菜单和前端交互

### 模式 B：最小真实链路备环境

特点：

- 使用真实后端
- 使用最小测试数据集
- 只验证某一条关键链路

适用场景：

- 主环境数据已污染
- 某业务闭环过长，不适合全量回归
- 只想验证一个模块的真实状态流转

推荐做法：

- 为每条链路准备最小 SQL fixture
- 为每条链路准备独立脚本
- 使用独立的测试月份、测试身份证、测试批次号前缀

## 3.3 主备切换原则

建议团队统一遵循：

1. 优先跑主测试环境
2. 主环境不可用时，不停止验证，立即切到备环境
3. UI 问题先在备环境复现，再回主环境确认
4. 任何通过备环境发现的问题，最终都要回主环境复核一次

## 4. 当前项目的自动化资产盘点

## 4.1 后端构建层

常用命令：

```powershell
mvn compile -DskipTests
```

生成可运行包：

```powershell
mvn -pl ruoyi-admin -am package "-Dmaven.test.skip=true"
```

说明：

- 当前仓库存在部分历史测试类未完全同步到最新业务代码
- 因此在“先验证业务链路，再回补测试类”场景下，允许暂时使用 `-Dmaven.test.skip=true`
- 团队目标仍应是最终恢复完整测试编译通过

## 4.2 接口与数据层

当前已落地脚本：

- `tests/导入预到龄测试数据.js`
- `tests/校验预到龄通知接口.js`
- `tests/验证预到龄通知闭环.js`
- `tests/预到龄发放通知测试数据.sql`
- `tests/预到龄通知闭环测试数据.sql`

共同特点：

- 从项目根目录 `.env` 读取数据库连接
- 使用 `sql/node_modules/dotenv` 和 `sql/node_modules/mysql2`
- 直接访问真实后端接口
- 自动准备数据、登录账号、推进状态、回查数据库
- 输出 JSON，适合人工阅读和 CI 解析

## 4.3 UI 自动化层

当前项目已具备 Playwright 基础能力：

- 配置文件：`ruoyi-ui/playwright.config.js`
- 当前用例：`ruoyi-ui/e2e/login-flow.spec.js`
- npm 脚本：`cd ruoyi-ui && npm run test:e2e`

当前配置现状：

- `baseURL` 默认值为 `http://127.0.0.1:83`
- 浏览器优先复用本机安装的 Chrome 或 Edge
- 当前已落地用例以 mock 登录和菜单权限验证为主

这说明项目已经有 Playwright 基础，但还没有形成完整的“真实业务 UI 自动化体系”。

同时，当前项目已经验证可使用 `Cursor 内置浏览器` 完成真实登录、真实页面进入、列表过滤和关键操作补证，因此后续 UI 测试计划应统一写成：

- `Playwright 自动化回归`
- `Cursor 内置浏览器真实页面补证`

## 7. Playwright UI 自动化准备说明

这部分是本文重点。目标不是“会写一个 Playwright 脚本”，而是让团队知道如何用 Playwright 尽可能模拟真实用户，并形成完整、可复跑、可审计的 UI 自动化体系。

补充说明：

- `Playwright` 不是唯一 UI 验证手段
- 对于真实环境联调，`Cursor 内置浏览器` 也是测试计划中的正式组成部分
- 在测试工作计划中，建议把 UI 验证拆成：
  - `Playwright 自动化`
  - `Cursor 内置浏览器补证`

## 7.1 UI 自动化的目标边界

Playwright 可以非常有效地模拟真实用户，但它无法凭空“保证不遗漏所有场景”。真正的不遗漏，必须来自：

- 完整的用例清单
- 清晰的状态机
- 明确的角色矩阵
- 边界与异常设计
- 自动化执行覆盖

也就是说：

- `Playwright` 负责执行
- `测试设计` 负责不遗漏

正确做法不是“写一条长脚本从头点到尾”，而是“先建覆盖模型，再让 Playwright 按模型执行”。

## 7.2 UI 自动化执行前提

在写任何业务 UI 自动化前，必须先准备：

- 稳定的测试账号
- 可重复清理的数据
- 稳定的页面选择器
- 可预期的环境地址
- 明确的前置数据构造方式
- 失败时可回放的 Trace/Screenshot/Video

## 7.3 页面选择器规范

团队应优先使用稳定选择器，而不是依赖样式类名。

推荐优先级：

1. `data-testid`
2. 语义化按钮文本
3. 表单 label
4. 明确的角色定位
5. 最后才是 CSS 结构选择器

最佳实践：

- 为关键按钮、输入框、表格操作列补充 `data-testid`
- 为弹窗确认、提交、审核通过、审核驳回等动作增加稳定标识

不推荐：

- 过度依赖 `.el-button--primary:nth-child(2)` 这类脆弱选择器

## 7.4 Playwright 如何模拟“像人一样”操作

正确理解“模拟人类”不是随机慢速点击，而是模拟真实用户的决策路径和页面等待行为。

推荐实践：

- 登录后等待首页关键元素出现，而不是固定 `sleep`
- 点击前先确认按钮可见、可用
- 对表格搜索先填写，再点击，再等待结果刷新
- 对弹窗先等待出现，再填写，再提交，再等待关闭
- 对上传操作等待接口返回和页面状态变化
- 对多角色流程使用多个 `storageState` 或分别登录

正确示例思路：

- 经办人登录
- 新建一条记录
- 提交
- 退出或切换角色
- 复核人登录
- 搜索刚才的记录
- 通过或驳回
- 审批人继续处理
- 最后用页面断言和接口断言双重确认

不推荐做法：

- 写一堆固定 `waitForTimeout(5000)`
- 每一步只断言“页面没报错”
- 只点页面不核对最终状态

## 7.5 UI 自动化覆盖模型

如果希望“不遗漏”，团队必须先按以下维度建模。

### 维度 A：角色覆盖

例如：

- 经办人
- 复核人
- 审批人
- 财务人员
- 审计人员

### 维度 B：状态覆盖

例如：

- 草稿
- 待提交
- 待复核
- 待审批
- 已通过
- 已驳回
- 已生成支付计划

### 维度 C：行为覆盖

例如：

- 新增
- 编辑
- 删除
- 查询
- 导出
- 导入
- 提交
- 驳回
- 批量审核
- 上传附件
- 下载模板

### 维度 D：异常覆盖

例如：

- 必填项缺失
- 身份证格式错误
- 重复数据
- 权限不足
- 上传错误文件
- 批量导入部分成功部分失败
- 后端超时或接口失败

### 维度 E：边界覆盖

例如：

- 空列表
- 单条记录
- 多条记录批量处理
- 极长文本
- 特殊字符
- 大文件上传
- 跨月、跨批次、重复提交

只有把这五个维度组合成矩阵，Playwright 的“完整测试”才有基础。

## 7.6 Playwright 用例分层

推荐拆成三层，而不是一个超长 spec：

### 第一层：页面冒烟

目标：

- 页面能打开
- 关键控件存在
- 基本按钮可点击

### 第二层：流程用例

目标：

- 完整用户路径
- 单角色或多角色流程
- 覆盖主要 happy path

### 第三层：异常与边界

目标：

- 各类校验提示
- 权限限制
- 异常回滚
- 导入导出错误处理

建议目录：

- `ruoyi-ui/e2e/smoke/*.spec.js`
- `ruoyi-ui/e2e/flows/*.spec.js`
- `ruoyi-ui/e2e/exceptions/*.spec.js`

## 7.7 UI 自动化前置数据准备

Playwright 要想稳定，必须避免“靠页面现造所有数据”。

推荐优先级：

1. SQL fixture 准备数据
2. Node 脚本调用接口准备数据
3. Playwright 页面仅负责验证交互和结果

原因：

- 页面造数据慢
- 页面造数据不稳定
- 一旦前置页面改版，所有后续用例都容易连锁失败

最佳实践：

- 复杂前置数据，用 `tests/*.sql` 或 `tests/*.js` 准备
- Playwright 只处理“用户真正关心的页面行为”

## 7.8 Playwright 真实后端模式与 Mock 模式

建议并行保留两种模式。

### Mock 模式

优点：

- 快
- 稳
- 不依赖后端
- 适合前端开发自测

当前项目示例：

- `ruoyi-ui/e2e/login-flow.spec.js`

### 真实后端模式

优点：

- 更接近生产
- 能发现联调问题
- 可验证真实权限、真实状态流转

缺点：

- 对环境依赖更高
- 数据清理更重要

建议：

- 页面结构和权限类用例优先 Mock
- 核心业务闭环用例必须有真实后端模式

## 7.9 Playwright 报告产出最佳实践

建议至少开启以下证据能力：

- 失败截图
- 失败 Trace
- 控制台日志
- 请求/响应失败日志
- 可选视频

团队目标不是“看控制台红字”，而是让失败可回放、可复盘、可转交。

建议后续将 `playwright.config.js` 扩展为：

- `reporter: [['html'], ['list']]`
- `use.trace: 'retain-on-failure'`
- `use.screenshot: 'only-on-failure'`
- `use.video: 'retain-on-failure'`

这样失败后，开发成员打开 HTML 报告即可看到：

- 哪条用例失败
- 失败步骤
- 失败时页面截图
- 每一步操作回放

## 8. 如何设计“完整而不遗漏”的自动化用例

不遗漏不是写更多脚本，而是先把需求拆成标准清单。

推荐每个模块都按以下结构建设用例。

## 8.1 用例清单模板

- `基础可访问`
  - 页面可打开
  - 权限正确
- `主流程`
  - 新增成功
  - 编辑成功
  - 提交成功
  - 审核成功
- `分支流程`
  - 驳回后重新提交
  - 部分通过
  - 批量处理
- `异常流程`
  - 缺字段
  - 错格式
  - 重复提交
  - 上传错误
- `边界流程`
  - 空数据
  - 单数据
  - 大批量
  - 特殊字符
- `权限流程`
  - 有权限可见可操作
  - 无权限不可见不可调用

## 8.2 用例唯一标识

建议所有用例引入固定编号，例如：

- `SMOKE-LOGIN-001`
- `FLOW-BENEFIT-003`
- `EXCEPTION-IMPORT-002`
- `AUTH-REVIEW-001`

这样报告、缺陷单、PR 讨论都可以直接引用编号。

## 8.3 预到龄示例

“预到龄通知”在本文中可作为标准示例链路：

1. 数据准备
2. 生成通知批次
3. 查看批次明细
4. 进入待遇核定
5. 录入身份证和银行卡
6. 上传 `用户ID.zip`
7. 提交审核
8. 复核人批量通过
9. 生成支付计划
10. 回查数据库与页面状态

这个例子说明的不是某个业务特殊性，而是一个通用原则：

- `接口脚本负责准备和核对`
- `Playwright 负责页面路径和用户交互`
- `报告负责沉淀结果`

## 9. 测试报告最佳实践

推荐所有自动化结果统一落为 Markdown 或 HTML 报告。

## 9.1 报告最小结构

每次执行至少输出：

- 测试名称
- 执行时间
- 代码版本
- 环境地址
- 数据版本
- 执行人或执行机
- 通过数
- 失败数
- 跳过数
- 失败明细
- 证据链接
- 风险结论

## 9.2 Markdown 报告模板

```md
# 自动化测试报告

## 基本信息
- 测试批次：`LOCAL-20260315-001`
- 分支/版本：`feature/xxx`
- 执行时间：`2026-03-15 21:00`
- 执行环境：`本地主测试环境`

## 执行结果
- 总用例数：30
- 通过：27
- 失败：2
- 跳过：1

## 失败明细
- `FLOW-BENEFIT-003`
  - 失败步骤：上传材料后未刷新状态
  - 证据：`playwright-report/...`
- `AUTH-REVIEW-001`
  - 失败步骤：无权限用户仍能看到按钮
  - 证据：`trace.zip`

## 风险结论
- 关键主流程：部分受阻
- 建议：修复权限问题后重跑审核链路
```

## 9.3 Playwright 报告最佳实践

建议：

- 控制台输出用于快速看结果
- HTML 报告用于团队查阅
- Trace 用于失败回放
- 截图用于缺陷单附件

## 9.3A Cursor 内置浏览器补证记录

当使用 `Cursor 内置浏览器` 做真实页面补证时，建议在测试报告中补充记录：

- 使用的账号
- 打开的真实页面 URL
- 看到的关键页面元素
- 验证成功的关键业务对象
  - 例如：通知批次号、核定记录、支付计划记录
- 如果有截图或页面快照，也应一并归档

## 9.4 接口脚本报告最佳实践

Node 自动化脚本建议输出两份结果：

- 标准 JSON
- 人类可读的 Markdown 汇总

JSON 适合机器解析，Markdown 适合团队沟通。

## 12. 当前方案的结论

当前项目最适合推广给团队成员的自动化测试体系是：

- 用 `Maven` 做构建层校验
- 用 `Node.js + SQL fixture + mysql2 + fetch` 做真实接口与数据库验证
- 用 `Playwright` 做 UI 自动化、角色权限验证、跨页面流程验证
- 用 `主测试环境 + 备测试环境` 保证验证连续性
- 用 `统一测试报告` 形成团队协作闭环

“预到龄通知”可以作为这套方案的标准示例，但这套方法本身适用于任何业务模块。

真正可复用的最佳实践不是某个脚本，而是下面这组组合能力：

- 有主备环境
- 有标准数据
- 有覆盖矩阵
- 有 UI 自动化
- 有接口自动化
- 有数据库核对
- 有统一报告

这才是团队级、本地可复用、可扩展到 CI 的完整自动化测试解决方案。
