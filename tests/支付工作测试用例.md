# 支付工作测试用例

**文档版本**: v1.0  
**创建日期**: 2026-03-15  
**测试对象**: 支付计划管理 + 财务管理 / 支付工作闭环

---

## 一、测试目标

本用例集用于验证“支付工作”是否能按设计形成完整闭环，并同步识别当前实现与目标设计之间的差距。

本次测试重点覆盖：

- 支付计划是否可生成并形成支付记录
- 支付批次是否可创建并进入复核、审批
- 财务是否可导出银行文件、提交银行、导入结果
- 发放失败是否可反馈、更正并发起二次/三次发放
- 页面、接口、权限、状态流转是否一致

## 四、前置条件

执行前需满足以下条件：

1. 前端、后端、数据库均已启动并可访问。
2. 至少存在一批已完成待遇核定、可进入支付计划的数据。
3. 测试账号已授予对应菜单和接口权限。
4. 已准备两类测试数据：
   - 正常发放数据
   - 故意失败数据，例如错误银行卡号或错误账户名
5. 已准备银行导出模板和银行结果导入样例文件。

## 六、执行方式

### 6.1 手工 UI 测试

- 经办人从“支付计划生成”开始走到失败处理
- 复核人、审批人、财务人员分别登录完成对应环节
- 每个环节记录页面现象、状态变化和提示信息

### 6.2 接口辅助验证

建议核对以下接口组：

- `/shebao/payment/plan/*`
- `/shebao/payment/review/*`
- `/shebao/payment/approve/*`
- `/shebao/payment/batch/*`
- `/shebao/finance/batch/*`
- `/shebao/finance/bank/*`
- `/shebao/finance/failure/*`

### 6.3 数据库辅助验证

重点核对：

- 支付记录是否新增
- 批次状态是否正确流转
- 银行结果导入后成功/失败数量是否回写
- 失败记录是否保留原因和重发次数

## 八、预期差距与风险口径

以下项即使在当前代码基线中复现，也应优先记录为“实现差距”，而不直接记为测试误操作：

- 前端已存在页面，但后端接口未完全对应
- 财务提交银行、导入结果返回固定成功消息或 `TODO`
- 失败处理页面动作与后端接口不一致
- 批次详情、重发追溯、成功失败汇总未完全落库

## 九、通过准则

### 9.1 当前实现通过

- 页面入口存在且可访问
- 支付计划基本查询/生成能力可执行
- 复核、审批基础状态流转可执行

### 9.2 闭环验收通过

只有满足以下条件，才可判定“支付工作闭环通过”：

- 支付计划生成成功
- 批次可创建并完成复核、审批
- 财务可导出、提交、导入结果
- 失败记录可更正并发起二次/三次发放
- 全链路状态与汇总统计可追溯
